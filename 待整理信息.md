# 企业级Agent开发标准流程

### 1. 场景定义与可行性分析 (Use Case & Feasibility)
*   **明确业务目标**：如智能客服、编写行程规划等。
*   **界定能力边界**：确定哪些由大模型处理，哪些由传统系统/规则兜底。
*   **定义输入与输出**：明确用户指令格式与结构化输出标准（如必须返回特定 JSON 字段）。

### 2. 基座模型选择与提示词工程 (Model Selection & Prompting)
*   **选型**：评估云端 API（GPT-4、Claude）或私有化部署开源模型（Llama 3、Qwen）的成本、延迟和推理能力。
*   **提示词**：编写系统提示词 (System Prompt)，通过角色扮演、示例 (Few-shot) 和思维链 (CoT) 来约束其行为。

### 3. 架构设计与编排 (Architecture & Orchestration)
*   **编排框架**：引入 LangChain、LangGraph、LlamaIndex 或企业级低代码平台（如 Dify、Coze、FastGPT）。
*   **流控制**：设计有向无环图 (DAG) 或状态机来管理工作流；决定使用单 Agent 模式（ReAct）还是多智能体（Multi-Agent）协作模式。

### 4. 知识库(RAG)与工具集成 (RAG & Tool Integration)
*   **RAG**：清洗相关数据，向量化并存入向量数据库，使 Agent 能动态检索最新知识（解决幻觉和信息滞后）。
*   **函数调用 (Function Calling/Tools)**：将外部系统封装为 API 供 Agent 调用，赋予其行动能力（如查询航班API、地图API）。

### 5. 评估基准与安全护栏 (Evaluation & Guardrails)
*   **评估 (Evals)**：构建覆盖核心场景的测试集。使用 LLM-as-a-Judge 或断言来量化测试 Agent 的准确率和工具调用成功率。
*   **安全护栏**：部署拦截器 (Guardrails)，防止 Prompt 注入及保证输出合规。

### 6. 部署、观测与迭代 (Deployment & Observability)
*   **部署**：服务容器化，利用流式输出 (Streaming) 优化响应体验。
*   **监控观测**：接入 LangSmith 等大模型日志监控工具，记录推理路径 (Trace)、Token 消耗和延迟。
*   **迭代**：收集人机交互反馈，提取 Bad Case 持续优化 prompt 和 RAG 检索策略。


# Agent 开发工具
*   **编排框架**：LangChain、LangGraph、LlamaIndex、CrewAI、Dify、Coze、FastGPT
*   **基座模型**：GPT-4、Claude、Llama 3、Qwen
*   **向量数据库**：Pinecone、Weaviate、Redis Vector Store
*   **评估工具**：LLM-as-a-Judge、Evals

其中：
- LangChain 是目前最成熟的 Agent 编排框架，提供了丰富的工具集成和灵活的工作流设计能力；
- LangGraph 则更适合复杂的多 Agent 协作场景，支持图形化设计和调试；
- LlamaIndex 则专注于知识库构建和 RAG 检索，适合需要大量外部知识支持的 Agent
- CrewAI 则提供了更轻量级的 Agent 编排方案，适合快速原型开发和小规模应用；
- Dify、Coze、FastGPT 等企业级低代码平台则提供了更友好的界面和部署方案，适合快速迭代和非技术团队使用。

一些开发tips：
1. 不要着急给用户最终的答案，通过多轮交互来获取更多的信息和偏向，在第一轮检索中先分析基本需求，然后抛出更多的问题
例如我跟gemeni的对话：5月10日去冰岛，5月19日离开，推荐每天的住宿，生成旅行规划。自驾
他会问我，酒店预算和是否喜欢温泉、徒步，摄影

2. 根据用户旅游行程的不同，给出的内容也不同，比如自驾应该是更注重路线规划和沿途景点，而非公共交通的时间表；如果用户喜欢温泉，应该优先推荐有温泉的住宿；如果喜欢徒步，应该推荐适合徒步的景点和住宿位置；如果喜欢摄影，应该推荐风景优美、适合拍照的地点和住宿。

3. 如果用户完全对目的地不了解，就应该先生成一些推荐选项，多做几轮交互以后再进行行程的生成
