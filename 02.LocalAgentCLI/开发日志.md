# OpenTravel 本地 CLI 开发日志

## 2026-04-18

### 起点

最开始的目标是做一个我自己能用、也适合面试展示的本地旅行规划 Agent。你一开始的要求很明确：不急着做前端，先把代码里的交互流程做出来，输入自己的旅行诉求后，能产出完整按天的攻略。

### 我最初的反馈

我先建议把这件事拆成本地 CLI 闭环，而不是先做前端页面。原因是你当前的资源和目标更适合先做：

- 输入前校验
- 模型生成
- 输出后校验
- 失败后局部修复
- 文本化结果导出

我也明确提醒了你，第一版最重要的是把 `slot-based JSON` 协议定下来，因为后面所有规则校验和展示都依赖它。

### 主要问题与原因

1. 你最初的输入里没有把“出发地 / 到达方式”单独列成必填项。
2. 直接把整趟行程一次性丢给小模型时，长输出非常容易超时。
3. 通过 LiteLLM 调 Ollama 时，`qwen3.5:4b` 会出现 `content` 为空的问题。
4. 使用 `qwen3.5:4b` 生成整段长 JSON 时，模型速度和稳定性都不够理想。

### 我们如何解决

1. 增加了输入前校验，把 `origin_city` 和 `arrival_mode` 设为必填。
2. 把主流程做成 CLI 本地版，先不依赖前端。
3. 建立了 `slot-based JSON` 结构，并在后端做时间冲突、字段完整性和 must-do 覆盖检查。
4. 改成直接调用 Ollama 原生 `/api/chat`，并加了 `think:false`，绕过 LiteLLM 的空内容兼容问题。
5. 引入 `daily` 分段生成模式，先生成骨架，再逐天生成并合并，提升小模型稳定性。
6. 给失败路径加了回退机制：某天失败时就用 scaffold 保底，不让整趟计划一起崩。

### 现在的结果

目前项目已经具备：

- 中文文档
- 中文注释
- 中文项目说明
- 中文开发日志
- 本地 Ollama 直连
- `qwen3.5:4b` 可运行
- `daily` 分段生成模式
- 输出 `plan.json` 和 `plan.txt`
- 终端交互编辑能力

### 现阶段的结论

这已经不是一个纯 prompt demo，而是一个可以继续演进的本地 Agent 原型。下一步最值得做的，是继续提升 `daily` 模式下的单日 prompt 质量，并逐步让 must-do 分配更稳定。
