# 项目结构说明

OpenTravel 本地 CLI 的结构是围绕“本地生成 + 代码校验 + 局部修改”来组织的，目标是先把产品主链路跑稳。

## 目录总览

```text
02.LocalAgentCLI/
  main.py
  README.md
  开发日志.md
  项目结构说明.md
  sample_request.json
  requirements.txt
  outputs/
  opentravel/
```

## 各部分职责

### `main.py`

命令行入口。负责：

- 读取输入 JSON
- 执行输入校验
- 选择生成模式
- 调用模型或 mock 生成器
- 执行输出校验
- 导出 JSON 和文本结果
- 可选进入交互编辑模式

### `opentravel/`

核心业务模块目录，里面每个文件分工比较清晰：

- `models.py`：配置、校验结果和常量定义
- `input_validation.py`：生成前的需求输入校验
- `planner.py`：行程生成调度器、骨架生成、按天生成逻辑
- `llm_client.py`：本地模型调用封装
- `plan_validation.py`：生成后的规则校验
- `refiner.py`：修正失败的输出
- `renderer.py`：把结构化 JSON 渲染成可读文本
- `editor.py`：终端交互修改 slot

### `sample_request.json`

示例输入文件，用来演示和测试当前的请求协议。

### `outputs/`

默认输出目录，保存运行结果：

- `plan.json`
- `plan.txt`

### `README.md`

项目说明文档，告诉你怎么安装、怎么运行、怎么切换模式。

### `开发日志.md`

记录这次项目从需求提出到现在的演进过程，方便你之后回顾，也方便面试时讲项目故事。

## 为什么这样组织

这个结构有几个好处：

1. 入口和业务拆开，CLI 保持简单。
2. 输入校验、生成、验证、渲染、编辑各自独立，后续容易替换。
3. `daily` 模式和 `whole` 模式都能共享同一套数据结构。
4. 后续如果要接前端，只需要复用 `opentravel/` 里的核心逻辑，不需要重写业务。

## 后续扩展方式

如果未来继续增强，这个结构也方便加新模块：

- 增加 `memory.py` 做用户长期偏好记忆
- 增加 `tools.py` 接地图、酒店、航班 API
- 增加 `evals.py` 做测试集和自动评估
