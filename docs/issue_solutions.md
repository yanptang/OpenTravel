格式：问题-原因-解决方案

问题一：通过 LiteLLM 调用 Ollama 时，`qwen3.5:4b` 出现过 `content` 为空的兼容问题
- 描述：模型在生成攻略时，直接返回了空白内容。
- 原因分析：LiteLLM 在处理 Ollama 的响应时，可能存在兼容性问题，导致无法正确解析模型的输出内容。并且 `qwen3.5:4b` 的think推理链过长，在接受了大量上下文后，可能会因为模型能力限制，context过载而无法正确生成内容。
> 超长推理链（CoT）模式，导致推理阶段耗尽显存/算力/缓存，最终无法输出结果;本地推理模型没有“推理链截断机制”，陷入推理坍塌，不停的循环，直到达到最大推理长度，最终输出空白。
- 解决方案：
  - 我取消了 LiteLLM 作为 Ollama 的主路径，改为直接调用 Ollama 原生的 `/api/chat` 接口
  - 禁止推理链输出：加上了 `think:false` 参数，避免过长的推理链导致的 context 过载或者显存爆炸问题。->有效
  - “指令精简”。尝试将长 Prompt 拆分成多轮对话：同时，我引入了 `daily` 分段生成模式，先生成攻略骨架，再逐天生成并合并，这样可以有效减少每次生成时的上下文量，避免模型被过多信息干扰。
- 最终结果：在 `qwen3.5:4b` 上，经过上述调整后，生成内容不再出现空白，且整体生成质量有了明显提升。 ->有效


问题二：语言本地化问题，输入中文，反馈的仍然是英文的旅游信息
- 原因：模型在处理多语言时，未能正确识别输入语言。而且后台调用prompt都是英文的，导致生成的内容默认是英文。
- 解决方案：根据输入的request的语言主要语言，增加了一个语言识别的步骤，把识别结果写回请求对象。然后在后续的prompt设计里都优先使用这个语言变量，这样就能保证生成内容的语言和输入一致了。


问题三：refine 不起效用
- 描述：把生成出来的攻略和识别出的错误再交给模型做一次修复，结果经常出现“看起来已经读过了，但最后还是原样返回”的情况。
- 原因分析：
  1. 输入上下文太重。原始整份攻略、完整错误列表、诊断报告、prompt 约束一起塞给模型，本地小模型容易被旧文本绑住，倾向保持自洽，而不是真正改动。
  2. 任务形式不对。让模型“修补整份 JSON”比“重新生成一版新攻略”更像结构化编辑，小模型在这种任务上往往更保守。
  3. 修复目标不够聚焦。即使知道有问题，如果没有明确指出只改哪一天、哪一个 slot、要改成什么样，模型还是容易选择最安全的原样输出。
- 解决方案重试：
  1. 先把整份修复改成按天局部重绘。validation 只负责找问题，refiner 只把有问题的那一天和必要上下文发给模型。
  2. 再把按天修复压缩成按单条问题修复。一次只让模型处理一个 primary issue，避免同一天里多个错误互相干扰。
  3. 给每个问题补充 issue_hint。比如酒店问题列出所有 hotel 候选，交通问题列出前后 slot，must-do 问题列出可落点候选。
  4. 把修复指令说得更明确。酒店问题要改成非 hotel，must-do 要写进相关 slot 的标题或详情，交通要写清起点和终点。
- 最终结果：
  - 采用了“按天 / 按问题局部重绘”的方案，而不是整份重写。
  - 结合更贴近真实旅行逻辑的 validation，模型终于开始真正改动内容。
  - 在 `qwen3.5:latest` 上，经过多轮局部修复后，原始计划的错误可以被逐步压到 0。
> 自洽性（Self-Consistency）： 模型在生成新内容时，会受到之前生成的文字的强烈干扰。如果之前的回复占了 1000 字，而你的修改指令只有 10 个字，在模型的注意力机制（Attention）里，那 1000 字的权重太大了，它很难跳出那个“逻辑怪圈”，导致了注意力的稀释： 本地模型（尤其是参数量较小的）存在 “中间失落” (Lost in the Middle) 现象。当对话变长，你让它修改时，它的 Attention（注意力）会被海量的旧文本分散，导致它更倾向于模仿自己刚才说过的“废话”，而不是执行你最新的“修改”指令
> 可能是模型能力不足，我从4b->9b了，结果还是没有变化，说明可能不是模型能力的问题

专业叫法：
1. 任务分治与编排（Orchestration）
我们不再把整份攻略和全部错误一次性发给模型，而是先由代码识别问题的类型、定位出错的 day/slot，再把局部问题路由到对应的修复 prompt。这样做的核心不是压缩上下文，而是主动管理模型的注意力范围。

2. 状态机迁移（State Transition）
我们把一份攻略视为一个可演化的状态。validation 负责判断当前状态是否存在问题，refiner 则在错误提示和局部上下文的引导下，生成一个新的局部状态，并替换回整份计划中。整个过程形成了“生成 -> 校验 -> 局部重绘 -> 再校验”的状态迁移闭环。
