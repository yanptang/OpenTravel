# OpenTravel 下一阶段开发计划

这个阶段我不再把目标放在“先接外部 API”或者“先做目的地知识库”上，而是先把攻略生成链路做成一个更像真实旅游管家的流程：

1. 先澄清关键需求。
2. 再把目的地活动偏好问清楚。
3. 再把景点、活动、交通、餐饮、酒店按时间线串起来。
4. 再把跨天闭环和输出文档规范化。

## 阶段目标

我希望这一阶段完成后，OpenTravel 至少能做到：

- 用户先输入一个粗需求，系统会主动追问，而不是直接胡乱生成。
- 目的地相关的本地活动偏好会先被挖掘出来，再进入行程生成。
- 行程输出不是“景点堆砌”，而是按天、按时间线、按交通衔接串起来的攻略。
- 酒店不只看当天终点，而是结合下一天路线做跨天闭环。
- 最终输出是一份可以直接阅读的 Markdown 攻略，而不是只有 JSON。

## 这一阶段的开发顺序

### 1. 多轮澄清

我先把澄清流程继续做实。

应该优先问的内容是：

- 出发地
- 目的地
- 出行日期
- 人数
- 到达方式
- 交通方式
- 必做事项

然后再问：

- 预算层级
- 旅行节奏
- 住宿偏好
- 当地特色活动偏好
- 特殊限制或偏好

这里的重点不是“多问”，而是“只问真正影响行程生成的内容”。

### 2. 时间线串联

我会把行程进一步做成时间线，而不是只输出 slot 列表。

每一天至少要包含：

- 交通衔接
- 活动安排
- 吃饭时间
- 酒店落点

如果是长线环岛或多日自驾，还要额外支持：

- 中途补给
- 路况提示
- 单日驾驶上限
- 体力分配建议

### 3. 跨天闭环

我会把“今天住哪里”和“明天去哪”绑定起来看，而不是只看当天结束点。

这一块的关键规则是：

- 到达日必须有明确的交通衔接
- 每天最后一个 slot 必须是住宿
- 第二天的第一段交通要和前一天住宿地衔接
- 住宿位置要尽量方便第二天的主线路

### 4. 输出文档化

我不想把 JSON 当最终产品形态。

JSON 继续作为中间态，负责：

- 机器校验
- 局部修改
- 后续记忆沉淀

真正给用户看的结果应该是 Markdown 文档，结构上更像一篇可直接使用的攻略。

### 5. 之后再接记忆和工具

等前面的链路稳定后，我再考虑：

- 用户偏好记忆
- 地图工具
- 酒店和航班工具
- RAG

这些能力都可以后置，不影响当前主线。

## 预期产物

这一阶段结束时，我希望至少有下面这些产物：

- 一个能进行多轮澄清的 CLI
- 一个能输出中文 Markdown 攻略的生成器
- 一个能做跨天闭环检查的 validator
- 一个能持续落盘的运行目录结构
- 一份可直接对外展示的攻略样例

## 这一阶段不做什么

我暂时不做：

- 真实航班和酒店实时查询
- 外部地图 API 接入
- 向量数据库 RAG
- 多 Agent 编排
- 前端页面

这些不是不重要，而是当前不是最优先项。
