# 攻略生成准则

这份准则来自我自己整理攻略的方式，以及我写过的冰岛攻略样例。它的作用不是描述某一次具体开发，而是定义 OpenTravel 以后“应该怎么写攻略”。

## 1. 先定关键输入

我在真正开始做攻略之前，必须先确认这些内容：

- 目的地
- 时间
- 人数
- 到达方式
- 出发地
- 交通方式

出发地虽然有时不是用户第一句就会说出来，但它决定了第一段交通和到达日怎么写，所以它必须被系统单独考虑。

## 2. 景点可以默认，活动要问清

我自己的经验是：

- 常见景点通常可以默认纳入行程
- 当地特色活动需要先问偏好

原因很简单：

- 景点通常是“去这个地方大概率会去”的
- 活动则更容易受个人偏好、预算和预订门槛影响

所以系统应该优先追问：

- 观鲸
- 冰川徒步
- 温泉
- 骑马
- 冰川洞穴探险

这类活动更适合先问，再安排。

## 3. 攻略必须按时间线写

我写攻略时不是先写景点列表，而是先把时间线串起来。

一个完整的 day 里面，我会关心：

- 几点出发
- 几点到达
- 玩多久
- 几点吃饭
- 几点去下一个点
- 晚上住哪里

这样写出来的内容才像一份实际能执行的攻略。

## 4. 酒店要看跨天闭环

我选酒店时不会只看当天的终点，还会看第二天去哪。

更合理的方式是：

- 看今天结束在哪里
- 看明天主线路从哪里开始
- 选一个夹在中间、方便衔接的位置

如果今天走到黄金圈，明天要去黑沙滩，那酒店不一定是当天终点附近最便宜的点，而是两天之间最顺的落点。

## 5. 每天都要有收尾

我写一整天的攻略时，结尾一定会落到：

- 酒店
- 休息
- 次日准备

这样整天的节奏才完整。

## 6. 最后要补实用信息

一份能用的攻略，不能只写“去哪儿玩”。

我通常还会补：

- 交通时间
- 路况提醒
- 是否需要提前预订
- 价格或预算
- 安全提示
- 穿衣建议
- 装备建议

## 7. 文档最终形态

我更希望最终输出是文档，而不是纯 JSON。

JSON 适合：

- 校验
- 调整
- 再生成

文档适合：

- 人直接阅读
- 分享给同行人
- 后续导出 PDF 或网页

## 8. 对 OpenTravel 的直接要求

OpenTravel 以后应该尽量满足下面这些要求：

- 先问清楚再生成
- 生成时默认景点，重点补活动、交通、酒店
- 输出必须是按天的时间线
- 跨天衔接要合理
- 语言要跟随用户输入
- 最终结果要能直接拿来用
