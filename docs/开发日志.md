# OpenTravel 本地 CLI 开发日志

## 2026-04-18

### 项目起点

我最初把这个项目的目标定为一个本地可用、适合面试展示的旅行规划 Agent 原型。第一阶段没有把重点放在前端页面，而是先把代码里的输入、生成、校验和输出闭环跑通。

我的核心诉求主要包括：

- 输入旅行需求后，能够生成完整的按天攻略
- 攻略内容需要结构化，便于后端校验
- 结果既能给人看，也能给程序继续处理

### 需求范围的调整

在项目推进过程中，我把输入协议补充了“出发地”和“到达方式”两个关键字段。这样可以把到达日的交通衔接纳入行程生成逻辑，避免第一天只从中午开始，而忽略出发城市到目的地之间的移动过程。

### 主要问题

1. 旅行需求如果缺少出发地或到达方式，第一天的行程结构就不完整。
2. 一次性生成整趟长行程时，小模型容易超时，也容易出现格式不稳定的问题。
3. 通过 LiteLLM 调用 Ollama 时，`qwen3.5:4b` 出现过 `content` 为空的兼容问题。
4. 直接让 `qwen3.5:4b` 生成长 JSON 时，速度和稳定性都不够理想。

### 处理方式

1. 我增加了输入前校验，将 `origin_city` 和 `arrival_mode` 设为必填项。
2. 我把主流程收敛为本地 CLI 版本，先不依赖前端页面。
3. 我统一使用 `slot-based JSON` 作为输出协议，并在后端增加时间冲突、结构完整性和 `must-do` 覆盖检查。
4. 我取消 LiteLLM 作为 Ollama 主路径，改为直接调用 Ollama 原生 `/api/chat`，并加上 `think:false`。
5. 我引入 `daily` 分段生成模式，先生成骨架，再逐天生成并合并。
6. 我为失败场景补充 scaffold 回退逻辑，避免单日失败拖垮整份计划。

### 当前结果

当前项目已经具备以下能力：

- 中文文档
- 中文注释
- 中文项目说明
- 中文开发日志
- 本地 Ollama 直连
- `qwen3.5:4b` 可运行
- `daily` 分段生成模式
- 输出 `plan.json` 和 `plan.txt`
- 终端交互编辑能力

## 2026-04-19

### 多轮澄清

我把需求澄清层接进了主流程，并且把澄清顺序拆成了三层：

1. 先补基础信息
2. 再基于目的地生成本地特色活动候选
3. 最后细化预算、节奏和住宿偏好

我做这一层不是为了“多问”，而是为了让系统更像一个真正的旅行管家。

### 目的地驱动的活动追问

我不再只靠固定规则问自驾、城市、住宿这些通用问题，而是先让模型根据目的地生成活动候选，再拿这些候选去追问用户。

例如目的地是冰岛时，优先出现的候选会更偏向：

- 观鲸
- 冰川徒步
- 骑马
- 温泉
- 海鹦观赏

### 中文输入和 Markdown 输出

我继续补了语言链路，让系统先根据输入内容判断语言，再把结果写回请求对象。这样后续的生成和渲染都能跟随输入语言。

我还把输出层改成了 Markdown 优先：

- `plan.json` 继续作为机器可读中间态
- `plan.md` 作为最终给人的攻略文档
- 如果需要，也可以切回纯文本 `plan.txt`

### 中文样例测试

我新增了一个天津到北京的中文两日游样例输入，用来验证：

- 中文输入
- 中文澄清
- 中文攻略
- Markdown 输出
- 独立运行目录，不覆盖历史文件

这次我把样例结果输出到单独目录里，而不是写回原来的默认产物。

### 当前结果

现在这套本地原型已经不只是“生成行程”，而是形成了一个更完整的前端无关链路：

- 先澄清需求
- 再生成时间线式行程
- 再做规则校验
- 再允许人工编辑
- 最后导出结构化和可读结果

### 当前结论

目前的实现已经从“prompt demo”推进为“可继续演进的本地 Agent 原型”。下一步最值得继续加强的方向，是提高 `daily` 模式下的单日生成质量，并让 `must-do` 的分配更稳定。

### 我发现的具体问题

这版北京攻略虽然整体可用，但还有几个很明确的细节问题：

1. **公共交通场景下，带行李先游玩不合理**
   - 用户如果选择的是公共交通，第一件事通常应该是去酒店放行李，或者先找地方寄存行李。
   - 目前的行程里，虽然写了入住，但行李处理和游览顺序还不够自然。

2. **景点之间的衔接还不够真实**
   - 比如故宫之后接王府井、再接其他地点，这种地点之间的移动还需要更细的路程和时间意识。
   - 现在能看出“有交通”，但还没有形成真正顺畅的路线串联。

3. **单个景点时长还要再校准**
   - 比如国博、故宫这类景点，时间通常不能只按模板估算，需要更符合实际游览节奏。

4. **重复景点问题**
   - 这版第二天又安排了故宫，说明当前生成逻辑还没有很好地避免“同一个核心景点重复出现”。
   - 这一点需要在下一阶段作为明确的约束来处理。

### 我的判断

这次问题不是“输出失败”，而是“行程质量还没完全达标”。  
从项目推进角度看，这其实是一个正常阶段，说明系统已经进入到可以细调规则和路线质量的阶段了。

### 明天要看的解决方向

我先把思路记下来，明天再具体处理：

- 公共交通优先补行李处理逻辑
- 景点之间补更细的地理衔接
- 核心景点时长按城市模板再校准
- 加一个“同一景点避免重复”的约束

这一轮我先不改代码，只把问题和后续方向记录下来，避免第二天再从头想一遍。

### 关于 RAG 的判断

我也重新想了一下这几个问题到底属于什么层次。

像“公共交通场景下先放行李”“景点之间要考虑衔接”“同一核心景点不要重复出现”这类内容，严格来说**不属于 RAG**，更像是：

- 行程规划规则
- 路线衔接约束
- 生成策略里的经验知识

如果把它们放到工程实现里，我更应该把它们放在：

- `planner` 的规划规则
- `validator` 的校验规则
- 未来的 `memory` 或策略层

RAG 更适合处理的是“外部事实”，比如：

- 景点开放时间
- 酒店可订性
- 具体路线信息
- 用户历史偏好以外的外部攻略材料

所以这部分我现在先按“规则层 / 经验层”来处理，而不是先往 RAG 里塞。

## 2026-04-21 补充记录：修复链路正式跑通

今天最关键的进展，不是又新增了一个功能点，而是把“生成后如何修”这条链路真正跑通了。

### 当前主线重新确认

- `whole` 模式仍然是当前主生成主线
- `daily` 模式后续还要继续做，但暂时不再承担当前阶段的主质量目标
- 当前重点是先把首版攻略质量、校验质量和修复闭环做稳

### validation / refiner 的职责变化

现在这两个模块的分工已经清楚很多：

- `validation` 负责发现问题、定位问题、标记问题类型
- `refiner` 负责根据问题做局部重绘
- 规则不再直接改写攻略正文，正文仍然由模型产出

当前实际运行链路已经更接近：

`生成 -> 校验 -> 局部重绘 -> 再校验`

### 为什么之前 refine 总失败

之前失败的核心原因主要是两点：

- 一次性把整份 plan、整份错误和整份诊断都喂给模型，本地小模型很容易原样返回
- 修复目标不够聚焦，模型知道有问题，但不知道该优先改哪一块

4B 和 9B 的测试都说明了同一个问题：不只是模型大小，更是修复任务的组织方式出了问题。

### 当前已经采用的修复策略

现在 refiner 已经改成了“局部重绘”模式：

- 先按 `day` 定位问题
- 再按单条 `issue` 逐个修
- 只把相关的 day / slot / 邻近上下文发给模型
- 没问题的天不发给模型
- 每轮修完立刻重新校验

这意味着：

- 已经满意的部分尽量不动
- 不满意的部分可以继续修
- 修复范围可控，不再整份乱飞

### 当前实际测试结论

以 `衡阳 -> 长沙 3 日游` 的 `whole` 模式案例为例，修复链路现在已经能做到：

- 第一版先通过 `validation` 发现问题
- 进入 `refine pass 1`
- 局部重绘后重新校验
- 在当前规则下通过校验并导出结果

这说明我们已经从“refine 总失败”推进到了“refine 可以真正改动内容并通过校验”。

### 对 daily 的重新判断

`daily` 之后仍然要继续做，而且会继续是项目的重要能力。

但当前阶段，`daily` 的定位变成了：

- 不是主生成策略的第一优先级
- 而是未来重点优化“分段记忆、跨天衔接、低上下文压力”的方向

换句话说：

- 生成主线先以 `whole` 为主
- 修复链路已经采用了更像 `daily + slot` 的局部更新思路
- 后续再把这种修复层面的经验，反哺回 `daily` 的生成质量

### 轻量 RAG 补齐

我今天还加了第一版简单 RAG。

这版不是向量数据库路线，而是**本地知识库 + 关键词召回 + prompt 注入**，目的是先做出一个足够清楚、足够可讲、足够可对比的 baseline。

当前实现方式是：

1. 在 `LocalAgentCLI/knowledge/` 下维护本地知识片段
2. 根据 `destination / origin / must_do / notes` 做关键词召回
3. 选择前 `top_k` 个知识片段
4. 把召回结果写入 `retrieval_context.json`
5. 把知识片段注入 planner prompt

首版知识库覆盖了：

- `Iceland`
- `Beijing`
- `Changsha`

这版 RAG 的意义不在于“检索技术多高级”，而在于：

- 它让项目具备了“外部知识增强”这一层
- 它可以直接做 `with_rag vs no_rag` 对比实验
- 它让后续更强的检索实现有了稳定的挂载点

- 为什么先做规则校验和 repair
- 为什么需要评估，而不是只看单次输出
- 为什么会尝试 LangGraph
- 为什么先做轻量 RAG，而不是一开始上向量库
- 为什么当前瓶颈集中在 transport continuity

### 如果还要继续补，最值得做什么

我不建议现在继续横向加太多“新名词功能”，而更建议做少量高价值补强。

最值得补的 4 件事是：

1. 做一组 `with_rag vs no_rag` 对照评估  
   这样 RAG 就不是“加了”，而是“证明了有没有价值”。

2. 把 `transport continuity` 再往前推进一轮  
   因为当前评估已经明确告诉我，主要问题都卡在这里。

3. 补一张系统架构图  
   面试时非常加分，比口头解释快很多。

4. 准备一段 2 分钟演示路径  
   例如：输入样例 -> 生成结果 -> 校验错误 -> 评估报告 -> LangGraph spike -> RAG 检索片段。

### 当前结论

我现在的判断是：

> 第一阶段已经完成。项目已经从“能跑的本地原型”进入“能评估、能解释、能展示、能继续迭代”的状态。

后续如果继续投入时间，重点不应该是再堆很多功能名字，而是：

- 用评估证明每个新增模块有没有价值
- 针对最主要的错误类型继续打磨质量
- 把当前系统更清楚地表达出来
