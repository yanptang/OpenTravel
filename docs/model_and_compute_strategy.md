# 模型与算力策略

这份文档定义 OpenTravel 在不同机器上的使用方式，以及不同阶段应该如何分配本地开发资源和学校算力资源。

## 总体原则

我希望这个项目保持 `local-first`：

- 日常开发尽量在本地机器上完成
- 大模型验证和批量测试放到学校算力上做
- 代码结构保持一致，避免本地版和算力版分叉太多

这样做的好处是：

- 本地开发效率高
- 大实验可以放到更强资源上
- 结果更容易对比和回归

## 本地机器定位

本地机器负责日常开发和轻量验证。

适合做的事情：

- 改代码
- 调 prompt
- 调澄清流程
- 调 Markdown 输出
- 跑 2 天游或 3 天游的小样例
- 验证 `qwen3.5:4b` 的基础可用性

推荐策略：

- 使用 `daily` 分段生成
- 控制每轮澄清问题数量
- 控制每一天的 slot 数量
- 默认输出中文 Markdown
- 结果按运行批次落盘，不覆盖历史文件

## 学校算力定位

学校算力负责更重的模型验证、长行程测试和批量评估。

适合做的事情：

- 更大模型的 A/B 对比
- 长线环岛行程测试
- 多轮澄清的批量回归
- 生成质量评估
- 修复策略测试
- 偏好记忆和后续 RAG 预研

这套资源更适合做“质量验证”和“规模化实验”，不适合只拿来跑日常小改动。

## 推荐分工

### 本地机

本地机负责：

- 开发 `LocalAgentCLI`
- 修改 `renderer`
- 修改 `clarifier`
- 修改 `planner`
- 跑快速样例
- 看输出是否符合中文表达和时间线逻辑

### 学校算力

学校算力负责：

- 跑多个模型版本
- 对比不同模型的行程质量
- 做长上下文测试
- 做批量输入输出测试
- 测试更复杂的目的地和更长的行程

## 模型使用建议

### 基线模型

`qwen3.5:4b` 作为当前基线模型，适合：

- 中文规划
- slot 化输出
- daily 分段模式
- 轻量本地验证

### 质量提升模型

如果后续要提高生成质量，可以在学校算力上尝试：

- 更大的 Qwen 系列模型
- 同级别的其他开源模型

重点不是盲目换模型，而是看下面这些指标：

- 中文一致性
- must-do 覆盖率
- 时间线合理性
- 跨天闭环质量
- 输出稳定性
- 生成时延

## 资源策略

### 本地侧

本地侧优先保证：

- 模型常驻时间足够长
- 上下文不要过大
- 生成不要一次太长
- 澄清不要一次问太多

### 学校算力侧

学校算力优先保证：

- 批量实验效率
- 多模型并行对比
- 长行程测试
- 评估脚本跑通

## 当前版本的推荐默认配置

如果我现在只看 OpenTravel 这一个项目，我会默认：

- 本地开发：`qwen3.5:4b`
- 分段生成：`daily`
- 输出格式：`Markdown`
- 结果落盘：独立运行目录

如果后面在学校算力上做验证，再逐步放开：

- 更大的模型
- 更长的上下文
- 更复杂的测试集

## 结论

最合理的方式不是让一台机器承担所有任务，而是分层使用资源：

- 本地机器负责快速开发和小样例调试
- 学校算力负责模型质量、批量测试和更重的实验

这样既能保证开发节奏，也能保证项目质量。
