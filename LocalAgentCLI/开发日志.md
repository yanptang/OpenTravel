# OpenTravel 本地 CLI 开发日志

## 2026-04-18

### 项目起点

我最初把这个项目的目标定为一个本地可用、适合面试展示的旅行规划 Agent 原型。第一阶段没有把重点放在前端页面，而是先把代码里的输入、生成、校验和输出闭环跑通。

我的核心诉求主要包括：

- 输入旅行需求后，能够生成完整的按天攻略
- 攻略内容需要结构化，便于后端校验
- 结果既能给人看，也能给程序继续处理

### 需求范围的调整

在项目推进过程中，我把输入协议补充了“出发地”和“到达方式”两个关键字段。这样可以把到达日的交通衔接纳入行程生成逻辑，避免第一天只从中午开始，而忽略出发城市到目的地之间的移动过程。

### 主要问题

1. 旅行需求如果缺少出发地或到达方式，第一天的行程结构就不完整。
2. 一次性生成整趟长行程时，小模型容易超时，也容易出现格式不稳定的问题。
3. 通过 LiteLLM 调用 Ollama 时，`qwen3.5:4b` 出现过 `content` 为空的兼容问题。
4. 直接让 `qwen3.5:4b` 生成长 JSON 时，速度和稳定性都不够理想。

### 处理方式

1. 我增加了输入前校验，将 `origin_city` 和 `arrival_mode` 设为必填项。
2. 我把主流程收敛为本地 CLI 版本，先不依赖前端页面。
3. 我统一使用 `slot-based JSON` 作为输出协议，并在后端增加时间冲突、结构完整性和 `must-do` 覆盖检查。
4. 我取消 LiteLLM 作为 Ollama 主路径，改为直接调用 Ollama 原生 `/api/chat`，并加上 `think:false`。
5. 我引入 `daily` 分段生成模式，先生成骨架，再逐天生成并合并。
6. 我为失败场景补充 scaffold 回退逻辑，避免单日失败拖垮整份计划。

### 当前结果

当前项目已经具备以下能力：

- 中文文档
- 中文注释
- 中文项目说明
- 中文开发日志
- 本地 Ollama 直连
- `qwen3.5:4b` 可运行
- `daily` 分段生成模式
- 输出 `plan.json` 和 `plan.txt`
- 终端交互编辑能力

## 2026-04-19

### 多轮澄清

我把需求澄清层接进了主流程，并且把澄清顺序拆成了三层：

1. 先补基础信息
2. 再基于目的地生成本地特色活动候选
3. 最后细化预算、节奏和住宿偏好

我做这一层不是为了“多问”，而是为了让系统更像一个真正的旅行管家。

### 目的地驱动的活动追问

我不再只靠固定规则问自驾、城市、住宿这些通用问题，而是先让模型根据目的地生成活动候选，再拿这些候选去追问用户。

例如目的地是冰岛时，优先出现的候选会更偏向：

- 观鲸
- 冰川徒步
- 骑马
- 温泉
- 海鹦观赏

### 中文输入和 Markdown 输出

我继续补了语言链路，让系统先根据输入内容判断语言，再把结果写回请求对象。这样后续的生成和渲染都能跟随输入语言。

我还把输出层改成了 Markdown 优先：

- `plan.json` 继续作为机器可读中间态
- `plan.md` 作为最终给人的攻略文档
- 如果需要，也可以切回纯文本 `plan.txt`

### 中文样例测试

我新增了一个天津到北京的中文两日游样例输入，用来验证：

- 中文输入
- 中文澄清
- 中文攻略
- Markdown 输出
- 独立运行目录，不覆盖历史文件

这次我把样例结果输出到单独目录里，而不是写回原来的默认产物。

### 当前结果

现在这套本地原型已经不只是“生成行程”，而是形成了一个更完整的前端无关链路：

- 先澄清需求
- 再生成时间线式行程
- 再做规则校验
- 再允许人工编辑
- 最后导出结构化和可读结果

### 当前结论

目前的实现已经从“prompt demo”推进为“可继续演进的本地 Agent 原型”。下一步最值得继续加强的方向，是提高 `daily` 模式下的单日生成质量，并让 `must-do` 的分配更稳定。

## 2026-04-19 补记

### 最新运行结果观察

我今天又跑了一版天津到北京的两日游，本地模型版已经能稳定输出中文 Markdown 攻略，整体格式也符合预期，说明以下功能链路已经打通：

- 中文输入
- 中文输出
- 本地模型调用
- `daily` 分段生成
- 规则校验
- Markdown 导出
- 进度日志

这说明目前的主链路已经可以继续往“攻略质量”上推进，而不是再停留在“能不能跑通”这个层面。

### 我发现的具体问题

这版北京攻略虽然整体可用，但还有几个很明确的细节问题：

1. **公共交通场景下，带行李先游玩不合理**
   - 用户如果选择的是公共交通，第一件事通常应该是去酒店放行李，或者先找地方寄存行李。
   - 目前的行程里，虽然写了入住，但行李处理和游览顺序还不够自然。

2. **景点之间的衔接还不够真实**
   - 比如故宫之后接王府井、再接其他地点，这种地点之间的移动还需要更细的路程和时间意识。
   - 现在能看出“有交通”，但还没有形成真正顺畅的路线串联。

3. **单个景点时长还要再校准**
   - 比如国博、故宫这类景点，时间通常不能只按模板估算，需要更符合实际游览节奏。

4. **重复景点问题**
   - 这版第二天又安排了故宫，说明当前生成逻辑还没有很好地避免“同一个核心景点重复出现”。
   - 这一点需要在下一阶段作为明确的约束来处理。

### 我的判断

这次问题不是“输出失败”，而是“行程质量还没完全达标”。  
从项目推进角度看，这其实是一个正常阶段，说明系统已经进入到可以细调规则和路线质量的阶段了。

### 明天要看的解决方向

我先把思路记下来，明天再具体处理：

- 公共交通优先补行李处理逻辑
- 景点之间补更细的地理衔接
- 核心景点时长按城市模板再校准
- 加一个“同一景点避免重复”的约束

这一轮我先不改代码，只把问题和后续方向记录下来，避免第二天再从头想一遍。

### 关于 RAG 的判断

我也重新想了一下这几个问题到底属于什么层次。

像“公共交通场景下先放行李”“景点之间要考虑衔接”“同一核心景点不要重复出现”这类内容，严格来说**不属于 RAG**，更像是：

- 行程规划规则
- 路线衔接约束
- 生成策略里的经验知识

如果把它们放到工程实现里，我更应该把它们放在：

- `planner` 的规划规则
- `validator` 的校验规则
- 未来的 `memory` 或策略层

RAG 更适合处理的是“外部事实”，比如：

- 景点开放时间
- 酒店可订性
- 具体路线信息
- 用户历史偏好以外的外部攻略材料

所以这部分我现在先按“规则层 / 经验层”来处理，而不是先往 RAG 里塞。
