# 项目结构说明

OpenTravel 本地 CLI 的目录结构围绕“本地生成 + 代码校验 + 局部修改”组织，目标是先把主链路跑稳，再逐步扩展能力边界。

## 目录总览

```text
LocalAgentCLI/
  main.py
  README.md
  开发日志.md
  项目结构说明.md
  sample_request.json
  tianjin_beijing_request.json
  requirements.txt
  outputs/
    runs/
  opentravel/
```

## 各部分职责

### `main.py`

命令行入口，负责：

- 读取输入 JSON
- 自动识别语言
- 执行输入校验
- 选择生成模式
- 调用模型或 mock 生成器
- 执行输出校验
- 导出 JSON 和 Markdown / 文本结果
- 可选进入交互编辑模式

### `opentravel/`

核心业务模块目录，各文件职责如下：

- `models.py`：配置、校验结果和常量定义
- `input_validation.py`：生成前的需求输入校验
- `clarifier.py`：多轮澄清，先补齐基础信息，再问目的地活动和通用偏好
- `planner.py`：行程生成调度器、骨架生成、按天生成逻辑
- `llm_client.py`：本地模型调用封装
- `plan_validation.py`：生成后的规则校验
- `refiner.py`：修正失败的输出
- `renderer.py`：把结构化 JSON 渲染成 Markdown 或纯文本
- `editor.py`：终端交互修改 `slot`

### `sample_request.json`

通用示例输入文件，用于演示和测试当前请求协议。

### `tianjin_beijing_request.json`

中文样例输入文件，用于验证“中文输入 -> 中文输出 Markdown 攻略”的链路。

### `outputs/`

默认输出目录，保存运行结果。

#### `outputs/runs/`

按运行批次保存每次产物，目录中通常会包含：

- `request.json`
- `plan.json`
- `plan.md` 或 `plan.txt`

### `README.md`

项目说明文档，包含安装方式、运行方式、模式切换说明和当前能力。

### `开发日志.md`

记录项目从需求提出到当前实现阶段的演进过程，便于后续回顾和项目表达。

## 结构设计原因

这种组织方式有几个直接收益：

1. 入口和业务拆开，CLI 保持简单。
2. 输入校验、生成、验证、渲染、编辑各自独立，后续容易替换。
3. `daily` 模式和 `whole` 模式可以共享同一套数据结构。
4. 如果后续接入前端，只需要复用 `opentravel/` 里的核心逻辑，不需要重写业务层。
5. 按运行批次保存产物，避免覆盖历史结果，也方便回溯测试。

## 当前主线的分工

当前项目已经形成三条清晰的线：

- `main.py` 负责流程编排
- `opentravel/` 负责核心能力
- `outputs/runs/` 负责结果落盘

## 后续扩展方向

后续如果继续增强，这个结构也方便继续加模块：

- 增加 `memory.py` 做用户长期偏好记忆
- 增加 `tools.py` 接地图、酒店、航班 API
- 增加 `evals.py` 做测试集和自动评估
