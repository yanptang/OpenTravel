# OpenTravel 实施计划

## 目标

先做出一个最小可运行的 MVP，完成下面这条闭环：

用户输入旅行需求 -> 系统追问关键信息 -> 生成结构化行程 JSON -> 页面展示结果

这个版本先不追求“全自动订票”或“实时全量数据”，重点是把产品主链路跑通。

## 第一阶段 MVP 范围

### 要做

- 一个简单网页
- 一个需求输入表单
- 一个“信息不足时自动追问”的逻辑
- 一个“生成结构化 itinerary JSON”的后端接口
- 一个结果展示页面
- 基础错误处理

### 暂时不做

- 地图 API
- 航班和酒店实时价格
- RAG
- 多 Agent 编排
- 自动预算优化
- 用户登录系统

## 我们建议的技术方案

为了先把原型快速做出来，建议用：

- 前端：`Streamlit`
- 后端逻辑：同一个 Python 项目内实现
- 模型调用层：`LiteLLM`
- 输出协议：固定 JSON Schema

这样做的好处是开发速度快，适合你当前这个验证阶段。

## 核心模块拆分

### 1. Input Parser

职责：

- 接收用户输入
- 整理成统一结构
- 判断是否缺少关键字段

关键字段建议：

- 出发地
- 目的地
- 出行日期
- 天数
- 出行人数
- 预算
- 交通方式
- 住宿偏好
- 必去体验

### 2. Clarifier

职责：

- 当信息不足时，生成 3 到 5 个关键追问
- 不直接生成行程

触发条件示例：

- 没有预算
- 没有交通方式
- 没有住宿风格
- 没有旅行偏好

### 3. Planner

职责：

- 根据完整需求生成结构化 itinerary
- 严格输出 JSON

### 4. Validator

职责：

- 校验 JSON 结构是否合法
- 校验时间是否冲突
- 校验必要字段是否齐全

第一版先做结构校验和基础逻辑校验，不做地图距离校验。

### 5. Renderer

职责：

- 把 JSON 渲染成用户能读懂的页面
- 至少展示：

- 每天日期
- 每个 slot 的时间
- 活动类型
- 标题
- 地点
- 说明

## 推荐开发顺序

### Step 1

先定义统一的数据结构，也就是 itinerary JSON Schema。

这是整个项目最重要的一步，因为后端输出、前端展示、后续校验都会依赖它。

### Step 2

做一个最简单的 Python 项目骨架。

建议目录：

```text
OpenTravel/
  app.py
  requirements.txt
  opentravel/
    __init__.py
    schema.py
    prompts.py
    planner.py
    validator.py
    ui.py
```

### Step 3

先不接复杂能力，只打通：

- 输入需求
- 返回模拟 JSON
- 页面展示

也就是说，先用 mock 数据把界面流程跑通。

### Step 4

把 mock planner 替换成真实 LLM 调用。

### Step 5

加入信息不足时的追问逻辑。

### Step 6

加入 validator，对模型输出做兜底。

## 本周最小里程碑

如果我们按最稳的节奏推进，第一个可交付里程碑可以定义为：

“在本地打开一个网页，输入旅行需求，点击按钮后，看到格式化展示的 3 天示例行程。”

只要这个做出来，项目就从“想法”正式进入“产品原型”阶段了。
